# Gemini Instructions for lon-stack-dx

## Project Overview

- **LON Stack DX** is a C-based protocol stack for building industrial IoT networks using the LON protocol defined in the ISO/IEC 14908 series of standards.
- The stack is modular, with clear separation between abstraction layers (hardware, OS, IP), protocol layers, and API entry points.
- Devices communicate via a publish-subscribe data model over native LON/IP over Ethernet or Wi-Fi, or native LON channels such as LON/FT or LON/TP-1250.

## Key Architectural Components

- **Abstraction Layers**: `abstraction/IzotCal.c`, `abstraction/IzotHal.c`, `abstraction/IzotOsal.c` provide connectivity, hardware, and OS abstraction.
- **API Entry Points**: `IzotApi.c` and `include/izot/IzotApi.h` expose the main stack API. Key functions include `IzotCreateStack()`, `IzotRegisterStaticDatapoint()`, `IzotStartStack()`, and `IzotEventPump()` for the stack lifecycle.
- **Protocol Layers**: Located under `lcs/`. The layers handle Application/Presentation (layers 7/6), Session/Transport (layers 5/4), Network (layer 3), and Data Link (layer 2) concerns.
- **LON/IP**: `lon_udp/ipv4_to_lon_udp.c` implements the LON/IP over Ethernet or Wi-Fi data link layer.
- **LON Native**: `lon_usb/lon_usb_link.c` implements the LON over USB data link layer using a U10 or U60 network interface with the MIP/U50 or MIP/U61 link-layer protocol.
- **Persistence**: `persistence/lon_persistence.c` and `persistence/storage_persistence.c` handle flash and persistent storage.

## Developer Workflows

- **Build**: The project uses CMake (`CMakeLists.txt`, `CMakePresets.json`). Source and header files are explicitly listed.
- **Stack Initialization**: A typical application should first call `IzotCreateStack()`, then register datapoints, and finally call `IzotStartStack()`. The main loop must recurrently call `IzotEventPump()`.
- **Datapoint (Network Variable) Handling**:
  - Declare a C variable and an `IzotDatapointDefinition`.
  - Register it using `IzotRegisterStaticDatapoint()`.
  - To send an update, modify the C variable and call `IzotPropagate()` or `IzotPropagateByIndex()`.
  - To receive updates, register a handler using `IzotDatapointUpdateOccurredRegistrar()`.
- **Application Messaging**:
  - Send messages with `IzotSendMsg()`.
  - Receive messages by registering a handler with `IzotMsgArrivedRegistrar()`.

## Project-Specific Conventions

- **Layered File Structure**: Each protocol layer is implemented in its own file(s) under the `lcs/` directory.
- **Interlayer Queues**: Data is passed between layers and between the link layer and hardware via queues using functions defined in `lcs/lcs_queue.c` and `include/lcs/lcs_queue.h`.
- **Explicit Handler Registration**: All event handlers (for datapoints, messages, etc.) must be registered with the stack using the provided registrar functions as described in `IzotApi.c`.
- **OS Portability**: Use the macros defined in `include/abstraction/IzotConfig.h` (e.g., `OS_IS(LINUX)`, `OS_IS(FREERTOS)`) to isolate OS-specific code. All OS-dependent logic should be implemented within the abstraction layer, primarily in `abstraction/IzotOsal.c`.
- **Timers**: Timers are managed using functions defined in `lcs/lcs_timer.c` and `include/lcs/lcs_timer.h`.
- **Include Style**: Use `<...>` for all includes, as include paths are managed by CMake in `target_include_directories`. Public headers are in `include/`.

## Key Files & Directories

- `IzotApi.c`: Core LON stack API implementation.
- `lcs/`: Main LON protocol stack implementation files.
- `lon_udp/`: LON-over-UDP (LON/IP over Ethernet or Wi-Fi) implementation.
- `lon_usb/`: LON-over-USB (LON over U10 or U60 with MIP/U50 or MIP/U61 LLP) implementation.
- `abstraction/`: OS, connectivity, and hardware abstraction layers.
- `persistence/`: Data persistence logic.
- `include/`: Public header files for the stack.
- `example/`: Example applications.
- `test/`: Test script.
- `CMakeLists.txt`: Main build configuration file.
- `README.md`: General project information.
